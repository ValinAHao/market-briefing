# Playwright Issues in gh-aw v0.43.23

## Summary

When using `tools: playwright:` in gh-aw agentic workflows, browser navigation to external domains fails even when those domains are correctly listed in the `network: allowed:` allowlist.

---

## Symptoms

All `browser_navigate` calls return:

```
ERR_BLOCKED_BY_CLIENT
```

This is a **Chromium-level** error, distinct from a proxy-level block.

---

## Root Cause

The gh-aw sandbox launches the Playwright MCP container with `--allowed-hosts` hardcoded to:

```
localhost, localhost:*, 127.0.0.1, 127.0.0.1:*
```

Despite domains being correctly listed in `network: allowed:` (which configures the Squid proxy allowlist), the `--allowed-hosts` flag in the Playwright container is **not updated** to include those domains. The Squid proxy logs confirm the custom domains are allowed at the proxy level, but the browser itself blocks the navigation before the request ever reaches the proxy.

Confirmed from logs:

```
[INFO] Allowed domains: ... wallstreetcn.com   ‚Üê Squid proxy: ‚úÖ allowed
browser_navigate("https://wallstreetcn.com/") ‚Üí ERR_BLOCKED_BY_CLIENT  ‚Üê Chromium: ‚ùå blocked
```

---

## What Was Tried

| Attempt | Result |
|---|---|
| `tools: playwright:` + `network: allowed: [wallstreetcn.com]` | `ERR_BLOCKED_BY_CLIENT` |
| `strict: false` + domains in allowlist | Same error |
| Adding `playwright` to the defaults network group | Same error |

---

## Documentation vs Reality

The [gh-aw reference docs](https://github.github.com/gh-aw/reference/playwright/) state that `network: allowed:` controls Playwright domain access. In practice, **this does not work in v0.43.23** ‚Äî the `--allowed-hosts` flag passed to the Playwright MCP container is not populated from the allowlist config.

This appears to be a **version-specific bug** in gh-aw v0.43.23. Newer versions may fix it.

---

## Workaround

Replace `tools: playwright:` with `tools: web-fetch:` and `bash: ["curl:*"]`.

These tools make plain HTTP requests that correctly go through the Squid proxy. Domains listed in `network: allowed:` work as expected.

**Additional fix required:** The copilot CLI also needs `--allow-all-urls` added to its command in the compiled lock file (`.github/workflows/morning-briefing.lock.yml`). Without it, `web_fetch` and `curl` fail with:

```
Permission denied and could not request permission from user
```

This flag is not generated by `gh aw compile` ‚Äî it must be added manually after compiling.

**Limitation:** `web-fetch` and `curl` cannot render JavaScript-heavy pages. If a target site requires JS execution, Playwright would be needed ‚Äî but remains broken in this gh-aw version.
